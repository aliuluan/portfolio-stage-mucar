Savoir-faire Techniques

Cette section décrit mes compétences en conception de bases de données et en développement d'architectures d'ingestion logicielles, appliquées à la consolidation des données du projet Mu-CAR.

Savoir-faire élémentaires couverts :

  • SF-BDD-1 : Traduire des règles métiers en contraintes d'intégrité physiques SQL.
  • SF-BDD-2 : Modéliser et structurer des relations et des clés étrangères complexes.
  • SF-DEV-1 : Développer des points d'accès (endpoints) API REST d'extraction et d'agrégation.
  • SF-ARCHI-1 : Orchestrer et isoler des services applicatifs et de données via la conteneurisation Docker.

Trace 1 : Script DDL de robustesse de la base de données PostgreSQL

-- [POINT REMARQUABLE A : CONTRAINTE DE CAPACITÉ]
ALTER TABLE requete_trajet 
ADD CONSTRAINT chk_capacite_vehicule 
CHECK (nombre_passagers_reel <= capacite_maximale);

-- [POINT REMARQUABLE B : COHÉRENCE TEMPORELLE]
ALTER TABLE fenetre_horaire 
ADD CONSTRAINT chk_coherence_horaire 
CHECK (heure_debut <= heure_fin);
Légende Trace 1 : Extrait du script SQL physique (v0.3) traduisant les règles de l'algorithme GBFCA en barrières de sécurité physiques dans PostgreSQL.

Descriptif général & place dans le stage : La Trace 1 représente l'aboutissement de mon travail de conception de données. Au sein du projet Mu-CAR, son rôle est capital : elle garantit qu'aucune donnée corrompue (comme un véhicule en surcharge ou un horaire inversé) ne puisse être enregistrée par le système de covoiturage, sécurisant ainsi les calculs du moteur de recherche.

Utilisation des savoir-faire : J'ai mobilisé le savoir-faire SF-BDD-1 pour implémenter des garde-fous directement au niveau physique. Dans la Trace 1, le [POINT REMARQUABLE A] montre comment la règle métier "un conducteur ne peut pas prendre plus de passagers que sa capacité maximale" est rigidifiée par un CHECK. Le [POINT REMARQUABLE B] applique la même logique pour la cohérence chronologique des fenêtres horaires, sécurisant le mapping de la table requete.

Descriptif général & place dans le stage : Mon démonstrateur d'interface front-end n'étant plus le composant le plus avancé, j'ai réorienté mes efforts sur l'infrastructure. La Trace 2 montre l'architecture de MucarGen, une API backend développée cette semaine. Son but n'est pas de remplacer l'interface de visualisation existante (Mucar_viz), mais d'agir comme un aspirateur de données : elle extrait les flux simulés de Mucar_viz pour les injecter et les structurer proprement dans ma base PostgreSQL, afin d'alimenter de manière fiable le démonstrateur Mu-CAR.

Utilisation des savoir-faire : Pour l'extraction, j'ai mis en œuvre le savoir-faire SF-DEV-1 dans le dashController.js [POINT REMARQUABLE C], en écrivant des requêtes de correspondance et d'agrégation (SUM, COUNT sur la table ticket) pour générer les statistiques environnementales de la zone pilote de Belfort-Montbéliard. Enfin, pour garantir la portabilité totale de cet aspirateur de données au laboratoire, j'ai utilisé le savoir-faire SF-ARCHI-1 via le fichier docker-compose.yml [POINT REMARQUABLE D], qui déploie instantanément l'API Node et la base de données isolée dans un conteneur léger Alpine.

Bilan sur le savoir-faire général : Conception et Ingestion de données

Résumé des savoir-faire élémentaires associés : Ce savoir-faire général regroupe la traduction de contraintes métiers en SQL (SF-BDD-1), la structuration relationnelle (SF-BDD-2), le développement d'endpoints d'extraction (SF-DEV-1) et le déploiement conteneurisé (SF-ARCHI-1).

Contexte d'utilisation : Ces compétences ont été mobilisées pour concevoir l'arborescence physique de la base Mu-CAR_gen et coder l'API de transition MucarGen destinée à aspirer les données issues des briques de simulation du laboratoire.

Contexte d'apprentissage & Difficultés : La modélisation conceptuelle (MCD) et les bases du SQL ont été étudiées en cours à l'IUT. Cependant, la transition itérative vers un modèle physique (MPD) intégrant des contraintes de robustesse complexes pour le covoiturage dynamique a représenté une réelle difficulté d'application. Concernant le développement de l'API avec Express, la connexion par Pool (pg) et l'orchestration multi-conteneurs Docker, il s'agit d'un apprentissage réalisé en totale autonomie à la maison et durant le stage, le programme de BUT 2 ne couvrant pas encore la conteneurisation isolée de production.
